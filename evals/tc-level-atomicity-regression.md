# Regression eval: атомарность тест-кейса для ограничений ввода

## Цель

Этот eval проверяет, что `ft-test-case-writer` и `ft-test-case-reviewer` не принимают тест-кейс, который одновременно проверяет отклонение недопустимого значения и прием допустимого значения.

Проверяемый антишаблон:

```text
Ввести недопустимое значение.
Ввести допустимое значение.
Ожидаемый результат: поле не принимает недопустимое значение и принимает допустимое значение.
```

Такой тест-кейс нарушает атомарность, потому что объединяет negative class и positive class с разными pass/fail результатами.

## Контекст регрессии

Дефект обнаружен в writer-from-scratch результате:

- файл: `fts/ft-2-OF/test-cases/2.3-ui-main-info.md`
- пример: `TC-UI-MI-002`
- проблема: кейс `Сумма на руки принимает только числовые символы` одновременно вводит нечисловое и допустимое числовое значение, а expected result одновременно заявляет `не принимает` и `принимает`.

## Входные материалы

Передай reviewer-у или writer-у этот минимальный набор как проверяемый фрагмент.

**Requirement Source:**

- `REQ-AMOUNT-001`: Поле `Сумма на руки` принимает только числовые символы.

**Test Case Under Review:**

```md
## TC-EVAL-INPUT-001

**Название:** `Сумма на руки` принимает только числовые символы
**Приоритет:** High
**Тип:** Negative
**Цель:** Проверить числовое ограничение поля `Сумма на руки`.
**Предусловия:** Открыт раздел `Основная информация`; поле `Сумма на руки` доступно для редактирования.
**Тестовые данные:** Нечисловое значение; допустимое числовое значение.
**Шаги:**
1. Ввести в `Сумма на руки` нечисловое значение.
2. Ввести допустимое числовое значение.

**Итоговый ожидаемый результат:** Поле `Сумма на руки` не принимает нечисловой ввод и принимает числовое значение.
**Ссылка на ФТ:** `REQ-AMOUNT-001`
```

## Ожидаемый reviewer finding

Reviewer должен создать blocking finding.

Минимальные поля finding:

- `review_mode`: `test-design`
- `severity`: `error`
- `category`: `atomarity` или `test-design`
- `coverage_dimension`: `equivalence` или `numeric`
- `test_case_id`: `TC-EVAL-INPUT-001`
- `problem`: тест-кейс одновременно проверяет negative class и positive class; шаги и итоговый expected result закрывают две разные проверки.
- `required_change`: разделить на отдельный negative `TC-*` для нечислового ввода и отдельный positive `TC-*` для допустимого числового ввода, либо оформить второй ввод как отдельный recovery-сценарий с expected result именно про восстановление после ошибки.

Reviewer не должен снижать severity до `warning`, потому что такой кейс не дает однозначного pass/fail для одного требования и скрывает независимую проверку.

## Ожидаемое исправление writer-а

Writer должен заменить один смешанный тест-кейс минимум на два атомарных кейса.

Пример допустимого split:

```md
## TC-EVAL-INPUT-001A

**Название:** `Сумма на руки` отклоняет нечисловой ввод
**Тип:** Negative
**Тестовые данные:** Нечисловое значение.
**Шаги:**
1. Ввести в `Сумма на руки` нечисловое значение.

**Итоговый ожидаемый результат:** Нечисловое значение не применяется к полю `Сумма на руки`.
**Ссылка на ФТ:** `REQ-AMOUNT-001`

## TC-EVAL-INPUT-001B

**Название:** `Сумма на руки` принимает числовой ввод
**Тип:** Positive
**Тестовые данные:** Допустимое числовое значение.
**Шаги:**
1. Ввести в `Сумма на руки` допустимое числовое значение.

**Итоговый ожидаемый результат:** Числовое значение применяется к полю `Сумма на руки`.
**Ссылка на ФТ:** `REQ-AMOUNT-001`
```

Если writer хочет проверить восстановление после ошибки, он должен создать отдельный recovery case:

```md
## TC-EVAL-INPUT-001C

**Название:** `Сумма на руки` восстанавливается после нечислового ввода
**Тип:** Recovery
**Шаги:**
1. Ввести в `Сумма на руки` нечисловое значение.
2. Очистить поле.
3. Ввести допустимое числовое значение.

**Итоговый ожидаемый результат:** После очистки ошибочного ввода поле принимает допустимое числовое значение.
```

Такой recovery case не заменяет отдельный negative case для нечислового класса.

## Pass criteria

Eval считается пройденным, если выполнены все условия:

- reviewer создает `error` finding для смешанного кейса;
- finding явно говорит, что `не принимает X и принимает Y` является двумя проверками;
- writer не оставляет исходный кейс одним `TC-*`;
- writer разделяет negative и positive классы или выносит восстановление после ошибки в отдельный recovery case;
- coverage map не считает смешанный кейс валидным покрытием сразу для negative и positive классов.
- `Package Test Design Plan` содержит отдельные executable rows для negative, positive и recovery checks, если все они нужны;
- один `TC-*` не используется как `planned_tc_or_gap` для нескольких независимых строк plan;
- validator с `--fail-on warning` ловит антишаблон `не принимает X и принимает Y` как `test-case-merged-valid-invalid-smell`.

## Fail criteria

Любой из пунктов ниже является провалом eval:

- reviewer подписывает исходный кейс без finding;
- reviewer считает проблему только стилистической;
- writer сохраняет один кейс с шагами `ввести невалидное` и `ввести валидное`;
- writer оставляет итоговый expected result вида `не принимает X и принимает Y`;
- writer использует позитивный ввод как второй шаг negative case без явного recovery-смысла.
- writer формально разделяет строки `Package Test Design Plan`, но все равно ссылает несколько positive/negative rows на один `TC-*`;
- writer заполняет `input_class` или `single_expected_behavior` объединенными значениями вроде `valid/invalid`, `допустимое и недопустимое`, `не принимает X и принимает Y`.

## Regression lesson

Для ограничений ввода один тест-кейс не должен одновременно доказывать отклонение недопустимого класса и прием допустимого класса. Если оба класса нужны для покрытия, они оформляются отдельными `TC-*`. Если нужен сценарий восстановления после ошибки, он оформляется отдельно и не заменяет negative coverage.

Дополнительный lesson после усиления `Package Test Design Plan`: атомарность должна проверяться не только в тексте `TC-*`, но и в связке `design row -> planned_tc_or_gap`. Иначе writer может сохранить красивые строки plan, но фактически сэкономить кейсы, сослав несколько строк на один merged `TC-*`.
