## Design Rules

- Один кейс проверяет один блок маппинга или одну decision table строку ФТ.
- Однотипные строки тегов внутри одного блока объединены, потому что выполняются одним действием генерации и имеют один основной ожидаемый результат: корректное заполнение блока печатной формы.
- BSR-кодов в разделе `4.4` нет, поэтому трассировка идет через `ATOM-*`, строку таблицы и источник.
- Не добавляются проверки UI-пути генерации, ролей, статусов, backend-событий и валидации исходных полей.
- Проверка точного имени тега использует DOCX как oracle, потому что XHTML/PDF содержат визуальные переносы; это зафиксировано в `coverage-gaps.md`.
- `Тип` в canonical TC использует только runtime enum `Positive` / `Negative`; decision-table характер проверки отражается в тестовых данных и design artifacts, а не в поле `Тип`.
- Конфигурационная привязка шаблона к типу тегов не оформляется executable TC без конкретного доступного артефакта настройки.
