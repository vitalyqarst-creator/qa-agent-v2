# DaData Reference Для AutoFin

## Статус И Назначение

- `reference_kind`: `external-vendor-reference`
- `package`: `AutoFin`
- `checked_at`: `2026-07-19`
- `authority_for_autofin_requirements`: `limited-user-confirmed`
- `derived_from`: официальная документация DaData, перечисленная в разделе «Источники»
- `governed_by`: `fts/AutoFin/source/PostFinal-v2/PostFinal-v2.docx`, подтверждённые support/clarifications и глобальный приоритет источников проекта

Документ описывает стандартные возможности DaData, необходимые для понимания интеграции, подготовки исполнимых UI-шагов и планирования UI-calibration. По `CLR-ADDR-004` официальный адресный контракт DaData является авторитетным динамическим источником значений адресных справочников AutoFin. Это ограниченное исключение не доказывает конкретную конфигурацию UI, trigger/debounce, тексты ошибок или другие продуктовые настройки AutoFin.

## Модель Источников

| Уровень | Что подтверждает | Что не подтверждает |
| --- | --- | --- |
| FT/support/approved clarification | Требуемое поведение AutoFin | Фактическую работоспособность текущего стенда |
| DaData vendor reference | Возможности и стандартный API-контракт DaData | Конфигурацию виджета, backend и UI AutoFin |
| Normal UI path evidence | Наблюдаемое поведение текущей версии AutoFin | Новое бизнес-требование, отсутствующее в ФТ |

При конфликте не выбирай удобный вариант молча. Фиксируй `mismatch-ft-ui`, дефект или coverage gap с точными ссылками на FT, vendor reference и UI evidence.

### Авторитетный Динамический Адресный Справочник

По подтверждению пользователя `CLR-ADDR-004` публичная документация DaData заменяет отсутствующий отдельный проектный справочник для адресных полей, прямо связанных с DaData в ФТ.

- `binding_type`: `external-dynamic-dictionary`;
- provider: `DaData`;
- официальный контракт: `https://confluence.hflabs.ru/pages/viewpage.action?pageId=222888017`;
- для поля региона: `from_bound=region`, `to_bound=region`;
- режим покрытия: динамическая зависимость, а не закрытый перечень всех значений;
- значения, приведённые в документации, допустимы только как representative fixtures и не считаются полным inventory;
- точное поведение AutoFin после ввода и выбора уточняется в интерфейсе и сверяется с ФТ.

Этот binding разрешает проверять, что поле использует возвращаемые DaData значения региона, и снимает blocker `missing-dictionary-values`. Он не разрешает приписывать AutoFin vendor defaults, которых нет в ФТ.

## Общая Модель Подсказок

Официальная документация описывает подсказки как помощь человеку при вводе:

1. Пользователь вводит часть значения.
2. Сервис возвращает массив `suggestions`.
3. Интерфейс показывает доступные варианты.
4. Пользователь выбирает конкретный вариант.
5. Интеграция может перенести `value`, `unrestricted_value` и отдельные поля `data` в форму.

Подсказки не предназначены для автоматической обработки базы и не гарантируют правильность предположения. Окончательный вариант выбирает человек. Для AutoFin это означает: тест не должен считать первый или единственный вариант автоматически правильным, если ФТ не задаёт иное.

Vendor API использует `POST`, JSON в UTF-8 и токен авторизации. Эти детали не являются UI-обязанностями AutoFin и не включаются в ручной UI expected result без отдельного наблюдаемого артефакта.

## Адресные Подсказки

### Возможности DaData

- Поиск работает по части адреса от региона до квартиры и по почтовому индексу.
- DaData поддерживает опечатки, неверную раскладку, исторические названия и синонимы.
- Частичное совпадение поддерживается в последнем слове запроса; это vendor behavior, а не гарантированная настройка AutoFin.
- Результат содержит отображаемый адрес `value`, полный адрес `unrestricted_value` и компоненты в `data`.
- Для России доступны компоненты от региона до квартиры, а также ФИАС/ГАР и КЛАДР identifiers.
- Выбранный адрес может быть разложен на гранулярные компоненты. В API выбор конкретной ранее предложенной подсказки описан запросом с `count = 1` и её `unrestricted_value`.
- Поддерживаются отдельные granular suggestions для региона, города, улицы и дома.

### Vendor Defaults И Ограничения

| Параметр/ограничение | Публичный контракт DaData | Правило для AutoFin |
| --- | --- | --- |
| `query` | Обязательный текст запроса, не более 300 символов | Не превращать лимит в UI expected result без FT/UI evidence |
| `count` | По умолчанию 10, максимум 20 | Не требовать точное число строк dropdown без UI evidence |
| `language` | По умолчанию русский | Не считать переключение языка функцией AutoFin без ФТ |
| `division` | По умолчанию административное; доступно муниципальное | Не выбирать модель деления без FT/observed binding |
| rate limit | До 30 запросов/с с одного IP | Не тестировать через UI как продуктовый лимит AutoFin |
| new connections | До 60 новых соединений/мин с одного IP | Не тестировать через UI без integration observability |

Vendor HTTP-коды включают `400`, `401`, `403`, `405`, `413`, `429` и `5xx`. Из них нельзя выводить тексты, fallback или retry интерфейса AutoFin.

### Подтверждённая Привязка AutoFin PostFinal-v2

| FT anchors | Подтверждённое поведение AutoFin |
| --- | --- |
| `BSR 116`, `BSR 141` | Адрес регистрации и фактический адрес интегрированы с DaData. |
| `BSR 117`, `BSR 142` | Обязательны регион и номер дома; отсутствие квартиры вызывает заданную красную подсветку и подсказку о квартире/частном доме. |
| `BSR 118`, `BSR 143` | Если адрес не найден в DaData, отображается `Некорректно указан адрес`. |
| `BSR 119`, `BSR 144` | Найденный адрес раскладывается по полям ручного ввода. |
| `BSR 120`, `BSR 145` | При ручном заполнении строка адреса собирается из перечисленных FT-компонентов в заданном порядке. |
| `BSR 324` | При заполнении через DaData в модели данных должно заполняться поле `kladr`; это внутреннее обязательство и не считается UI-проверенным без наблюдаемого evidence. |

Подтверждённое пользователем уточнение в `open-scope-coverage-gaps_ответы Соболева.md`: для AutoFin используется бинарная трактовка — адрес либо найден в DaData, либо не найден целиком; отдельное состояние «частично найден» не заявлено.

## Подсказки По ФИО

### Возможности DaData

- Подсказки работают для ФИО одной строкой и отдельно для фамилии, имени или отчества.
- Сервис может исправлять раскладку и определять пол.
- Ответ содержит `value`, `surname`, `name`, `patronymic`, `gender` и `qc`.
- Подсказки не гарантируют корректное автоматическое распознавание и требуют выбора человека.
- Vendor параметры включают `count` (по умолчанию 10, максимум 20), `gender` и `parts`.

### Подтверждённая Привязка AutoFin PostFinal-v2

- `BSR 49`, `BSR 52`, `BSR 55`, `BSR 69`, `BSR 73`, `BSR 77`: при интеграции DaData допускаются подсказки для соответствующих полей ФИО.
- `BSR 59`: поле пола обновляется данными DaData после заполнения ФИО через подсказку DaData.

Не выводи из vendor reference, что AutoFin обязан исправлять раскладку, показывать определённое число вариантов или автоматически принимать предложенный пол: это требует FT либо UI evidence.

## Подсказки «Кем Выдан Паспорт»

### Возможности DaData

- Полнотекстовый поиск работает по `code` и `name` подразделения.
- Возможна фильтрация по `region_code` и типу подразделения.
- Результат содержит отображаемое значение, код подразделения, название в форме «кем выдан», код региона и тип подразделения.

### Подтверждённая Привязка AutoFin PostFinal-v2

- `BSR 94`: поле предзаполняется после ввода кода подразделения.
- `BSR 95`: если DaData возвращает несколько значений, пользователь выбирает нужное из списка.

Не требуй конкретную фильтрацию, сортировку или число результатов без UI evidence.

## Подсказки По Организациям

### Возможности DaData

- DaData ищет организации и ИП по ИНН, ИНН/КПП, ОГРН, названию, ФИО ИП и адресу.
- Результат может содержать тип организации, реквизиты, ОПФ и статус.
- Набор возвращаемых полей зависит от метода, тарифа и полноты данных источника.

### Подтверждённая Привязка AutoFin PostFinal-v2

- `BSR 265`: поле организации интегрировано с DaData.
- `BSR 266`: для недействующей организации AutoFin показывает заданную красную подсказку и блокирует `Далее`; проверка действует только при доступном сервисе DaData.
- `BSR 268`: ОПФ заполняется после выбора организации по интеграции DaData.
- `BSR 325`: выбор организации из списка предзаполняет перечисленные в ФТ поля анкеты.

Для этого блока FT уже задаёт часть fallback-семантики через условие доступности сервиса. Не обобщай её на адреса, ФИО или паспортное подразделение.

## Что Требуется Уточнять В UI

Для каждого DaData-поля фиксируй отдельно:

- после какого фактического ввода появляется первый список подсказок;
- применяется ли debounce и виден ли пользовательский loading state;
- сколько вариантов отображается и меняется ли порядок;
- допускается ли клавиатурный выбор и выбор мышью;
- какое значение остаётся в основном поле после выбора;
- какие связанные поля заполняются или очищаются;
- что происходит при редактировании уже выбранного значения;
- что происходит при отсутствии подсказок;
- доступен ли ручной ввод и как он взаимодействует с результатом DaData;
- что наблюдает пользователь при недоступности сервиса;
- сохраняется ли выбранное/ручное значение после навигации и возврата.

Эти пункты — calibration questions, а не готовые expected results.

## Классификация UI Evidence

| Статус | Значение |
| --- | --- |
| `confirmed-ft` | Normal UI path подтверждает точное требование ФТ. |
| `observed-vendor-aligned` | Наблюдение соответствует vendor reference, но отдельного требования FT нет. |
| `mismatch-ft-ui` | UI расходится с FT; требуется дефект/решение, а не переписывание baseline. |
| `vendor-variation` | UI отличается от vendor default, но FT допускает конфигурацию. |
| `not-observed` | Поведение не удалось воспроизвести. |
| `blocked-observability` | Поведение внутреннее или недоступно через normal UI path. |
| `coverage-gap` | Для исполнимого ожидаемого результата недостаточно FT/clarification/evidence. |

Для `confirmed-ft` и `mismatch-ft-ui` применяй `references/agent/ui-evidence-policy.md`: normal UI path и screenshot обязательны; для mismatch сохраняй trace, если Playwright доступен.

## Запреты Для Test Design

- Не использовать публичную документацию DaData для изменения обязательности, редактируемости, видимости или продуктовых expected results AutoFin. Узкое исключение `CLR-ADDR-004`: для адресных полей с прямой привязкой к DaData официальный API-контракт является динамическим источником allowed values, но не закрытым статическим перечнем.
- Не придумывать trigger length, debounce, dropdown layout, сортировку, тексты ошибок, fallback и retry.
- Не считать API-запрос, `kladr`, persistence, RabbitMQ или изменение модели проверенными через UI без наблюдаемого артефакта.
- Не переносить правила одного типа подсказок на другой без FT/clarification.
- Не считать текущее UI-поведение нормативным, если оно противоречит FT.
