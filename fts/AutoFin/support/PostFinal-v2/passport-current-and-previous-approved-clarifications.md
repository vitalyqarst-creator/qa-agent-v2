## Правила Использования Ответов

- `CLR-PASS-001` исключает весь раздел 5 «Ограничения по типам данных» и следующий общий default `NULL` из testable assertions/atoms/obligations/TC проекта AutoFin.
- Исключение не распространяется на локальные ограничения `BSR 84–114`, прямо заданные для серии, номера, кода подразделения и дат выдачи паспорта.
- Исключённые global rows сохраняются в source inventory как `not-applicable / out-of-project`, но не становятся context dependencies целевых полей.
- `CLR-PASS-002` разрешает использовать официальный `fms_unit` contract DaData как `external-dynamic-dictionary` для `BSR 94–95`.
- Concrete fixture `FX-DADATA-FMS-POS-001` фиксирует запрос `772-053`, несколько фактически возвращённых вариантов и точное выбранное значение; fixture применяется в режиме `verified-once / revalidate-on-failure`.
- Уточнение не задаёт trigger length, debounce, сортировку, точное число строк UI dropdown, fallback/retry или тексты ошибок, отсутствующие в ФТ.
- `CLR-PASS-003` разрешает строить точные граничные сценарии BSR 100–101 с календарными днями и датой рождения клиента как входом; 29 февраля не использовать без отдельного правила leap-day.
- `CLR-PASS-004` ограничивает test design последним предыдущим паспортом и запрещает придумывать максимальное количество повторов.
- Расхождение BSR 107/109 по составу добавляемых и удаляемых полей не закрывается этими ответами: source reviewer должен сохранить его как coverage gap и не приписывать предыдущему паспорту поля, не определённые в его строках.
- Старые паспортные тест-кейсы и legacy handoff не являются source input нового writer/reviewer run.
