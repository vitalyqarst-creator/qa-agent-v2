## Scope Boundaries

Набор покрывает формирование и содержимое печатной формы `Заявление-анкета на получение потребительского кредита` по разделу `4.4`: шаблон, подстановку тегов и заполнение полей/чек-боксов в PDF по данным клиента из Системы.

Не покрываются: UI-путь запуска генерации, роли, статусы заявки, API/RabbitMQ/БД, валидация исходных полей, правила сохранения исходных данных и поведение раздела `4.3`, кроме использования его данных как уже подготовленной фикстуры.

## Source Notes

- В разделе `4.4` нет BSR-кодов. Трассировка ведется через `ATOM-PFG-*`, строку таблицы и источник.
- Приоритетный источник требований: `FT4AutoFinFinal.xhtml`.
- Значение шаблона `Анкета клиента 04.02.2026.pdf` подтверждено `FT4AutoFinFinal.pdf`, page 36; в XHTML соответствующая ячейка пустая.
- Точное написание тегов сверено по `FT4AutoFinFinal.docx`, table 9, column `Тег`, потому что XHTML/PDF содержат визуальные переносы тегов.
- Проверка конфигурационной привязки шаблона к типу тегов не является executable TC без конкретного доступного артефакта настройки; см. `GAP-006`.
- Проверка расчета `<job_start>` не является executable TC без раскрытого справочника `Стаж работы`; см. `GAP-004`.
- Writer-side split artifacts для source inventory, package plan, quality gate, self-check и review находятся в `fts/AutoFin/work/test-design/section-16-print-form-generation/`.

## TC-AF-PFG-001

**Название:** Автоматическое формирование заявления-анкеты по шаблону `Анкета клиента 04.02.2026.pdf`

**Тип:** Positive

**Приоритет:** High

**package_id:** WP-01

**Трассировка:** `ATOM-PFG-001`; `ATOM-PFG-002`; `4.4 Требования к формированию печатных форм документов`; PDF page 36

### Предусловия

- В Системе есть заявка по фикстуре `FIX-PFG-BASE`.
- Для тестового стенда определен entrypoint формирования печатной формы; проверка самого entrypoint не входит в scope.

### Тестовые данные

- Фикстура: `FIX-PFG-BASE`.
- Печатная форма: `Заявление-анкета на получение потребительского кредита`.
- Шаблон: `Анкета клиента 04.02.2026.pdf`.

### Шаги

1. Открыть подготовленную заявку `FIX-PFG-BASE`.
2. Запустить формирование печатной формы заявления-анкеты через entrypoint тестового стенда.
3. Открыть сформированный PDF.

### Итоговый ожидаемый результат

Сформирован PDF `Заявление-анкета на получение потребительского кредита` по шаблону `Анкета клиента 04.02.2026.pdf`, и данные клиента в PDF заполнены на основании данных заявки `FIX-PFG-BASE`.

### Постусловия

Сохранить сформированный PDF как evidence выполнения проверки.

## TC-AF-PFG-002

**Название:** Отсутствие незаполненных или неизвестных тегов после генерации PDF

**Тип:** Negative

**Приоритет:** High

**package_id:** WP-01

### Предусловия

- В Системе есть заявка по фикстуре `FIX-PFG-BASE`.
- Для всех применимых строк таблицы раздела `4.4` в заявке заполнены исходные значения из `FIX-PFG-BASE`.

### Тестовые данные

- Перечень ожидаемых тегов: `DICT-001`.
- Негативный oracle: в итоговом PDF не должно остаться плейсхолдеров формата `<...>`.

### Шаги

1. Сформировать печатную форму заявления-анкеты для заявки `FIX-PFG-BASE`.

### Итоговый ожидаемый результат

В сформированном PDF нет незаполненных плейсхолдеров из таблицы раздела `4.4` и нет неизвестных тегов, не входящих в `DICT-001`.

### Постусловия

Сохранить PDF и результат поиска плейсхолдеров как evidence.

## TC-AF-PFG-014

**Название:** Выбор адреса регистрации в блоке `Согласия на обработку ПДн`

**Тип:** Positive

**Приоритет:** High

**package_id:** WP-04

**Трассировка:** `ATOM-PFG-016`; `4.4 tag row 61`

### Предусловия

- Подготовлены заявки по фикстурам `FIX-PFG-ADDR-SAME` и `FIX-PFG-ADDR-DIFFERENT`.

### Тестовые данные

- `Тег 1`: `<region_with_type>`, `<city_district>`, `<city>`, `<street>`, `<house>`, `<block>`, `<str>`, `<flat>`.
- `Тег 2`: `<region_with_type_reg>`, `<city_district_reg>`, `<city_reg>`, `<street_reg>`, `<house_reg>`, `<block_reg>`, `<str_reg>`, `<flat_reg>`.

### Шаги

1. Для каждой адресной фикстуры сформировать печатную форму заявления-анкеты.
2. Открыть сформированный PDF.
3. Проверить адрес регистрации в блоке `Согласия на обработку ПДн`.

### Итоговый ожидаемый результат

Для `FIX-PFG-ADDR-SAME` в блоке согласий используется `Тег 1`; для `FIX-PFG-ADDR-DIFFERENT` используется `Тег 2`, где поля `Тег 2` соответствуют адресу регистрации из фикстуры.

### Постусловия

Не требуются.

## TC-AF-PFG-015

**Название:** Заполнение кода подразделения и ранее выданных паспортов

**Тип:** Positive

**Приоритет:** High

**package_id:** WP-04

**Трассировка:** `ATOM-PFG-016`; `4.4 tag rows 62-87`

### Предусловия

- В Системе есть заявка по фикстуре `FIX-PFG-PREV-PASSPORTS-5`.
- В заявке заполнен код подразделения действующего паспорта.

### Тестовые данные

- Тег действующего паспорта: `<dep_code>`.
- Наборы тегов ранее выданных паспортов: `<serial_number_1>`..`<serial_number_5>`, `<number_1>`..`<number_5>`, `<issue_date_1>`..`<issue_date_5>`, `<issue_place_1>`..`<issue_place_5>`, `<dep_code_1>`..`<dep_code_5>`.

### Шаги

1. Сформировать печатную форму заявления-анкеты для заявки `FIX-PFG-PREV-PASSPORTS-5`.
2. Открыть сформированный PDF.
3. Проверить код подразделения действующего паспорта и блоки ранее выданных паспортов 1-5.

### Итоговый ожидаемый результат

Тег `<dep_code>` заполнен кодом подразделения действующего паспорта, а каждый набор тегов ранее выданного паспорта 1-5 заполнен серией, номером, датой выдачи, органом выдачи и кодом подразделения соответствующего ранее выданного паспорта из `FIX-PFG-PREV-PASSPORTS-5`.

### Постусловия

Не требуются.
