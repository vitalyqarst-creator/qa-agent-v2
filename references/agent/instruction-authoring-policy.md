# Правила написания инструкций для агента

Документ регулирует agent instructions; размещение, дублирование и загрузку задают `content-placement.md`, `duplication-policy.md` и `deep-reference-loading-policy.md`.

## Область и достаточность

- H1 должен раскрывать назначение без знания имени файла.
- Ограничивай документ одной областью и знаниями, меняющими решение агента; остальное заменяй ссылкой на канонический контракт.
- Не добавляй пояснения ради неправдоподобной ошибки компетентного агента.

## Обобщение и размещение

- Описывай одинаковую семантику одним правилом на самом высоком переиспользуемом уровне, где оно остается точным.
- Разделяй правила только при различиях в применимости, авторитете, поведении, результате или обработке ошибки.
- В зависимых документах оставляй только локальные условия, дополнительные последствия и ссылку на канонический источник.
- Не обобщай правила разных источников авторитета, если это скрывает приоритет источника или меняет поведение при конфликте.
- Явную зависимость помечай `governed-by` (нормативный контракт), `validated-against` (проверка) или `derived-from` (источник); влияющий на решение `governed-by` target должен входить в фактический context.

## Структура и модальность

- Группируй требования по аспектам и логическим зависимостям. Заголовок или вводная фраза должны задавать область действия группы.
- Сохраняй элементы одного списка на одном семантическом уровне; уточнение вкладывай в относящийся к нему элемент.
- Используй нумерованный список только для последовательности, где порядок влияет на результат. Для неупорядоченного набора используй маркированный список.
- В пределах одного документа используй согласованную систему модальности. Не смешивай русские императивы с `MUST`, `SHOULD` и `MAY`, если документ не определяет значения этих терминов.

## Контракты других компонентов

- Описывай задачу инструмента, условия запуска, входы, выходы и критерий успешности.
- Считай `<script-path> --help` каноническим источником синтаксиса CLI, флагов и значений по умолчанию. Не воспроизводи полный интерфейс CLI в agent instructions.
- Допускай минимальную команду запуска или проверки, но не альтернативный справочник параметров.
- Храни lifecycle и handoff в управляющем workflow; локально описывай только собственные условия и результаты.

## Целевая модель

- Описывай актуальное целевое поведение напрямую.
- Упоминай историю только тогда, когда она меняет вход, хранение, совместимость, обработку ошибки или текущее решение агента.
- Не используй историческое сравнение вместо формулировки действующего правила.

## Ревью в контексте целевого агента

После изменения agent instructions:

1. Определи затронутые scenarios и получи их фактический контекст через `resolve_instruction_context.py`; CLI-контракт смотри в `scripts/resolve_instruction_context.py --help`.
2. Проверь материал как компетентный целевой агент, которому доступен только контекст выбранного сценария.
3. Исправь недостаточные, противоречивые, избыточные и дублируемые указания. Считай замечание существенным, если оно меняет источник, scope, маршрут, полномочия, результат, состояние, gate или обработку ошибки. Не удаляй guardrail, подтвержденный повторявшимся runtime-дефектом.
4. Повторяй проверку до закрытия существенных замечаний; стилистические предпочтения не учитывай, неразрешимое противоречие фиксируй как blocker. Затем запусти архитектурные и бюджетные gates.

Успешные lint, audit и budget checks необходимы, но не заменяют проверку смысловой достаточности ограниченного runtime-контекста.
