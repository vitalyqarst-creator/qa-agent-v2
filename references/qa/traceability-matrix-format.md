# Traceability Matrix Format

Канонический traceability matrix artifact хранится в Markdown и используется для проверки полноты покрытия утверждений ФТ тест-кейсами. Для каждой создаваемой traceability matrix обязательно создается дублирующая `.xlsx`-версия с теми же строками и колонками.

## Назначение

- разложить выбранный scope на атомарные утверждения;
- явно показать, какие утверждения покрыты тест-кейсами;
- зафиксировать `gap` и `unclear`, не додумывая поведение системы;
- передать writer-у точечные пробелы покрытия в revision mode.

## Гранулярность строк

- Одна строка матрицы = одно атомарное утверждение или одно отдельное правило из ФТ.
- Если одно поле описано несколькими независимыми свойствами, каждое свойство должно идти отдельной строкой.
- Если условие меняет поведение поля, условное правило должно быть отдельной строкой.

## Обязательные колонки

- `atom_id`
- `req_id`
- `source_path`
- `atomic_statement`
- `field_or_block`
- `condition`
- `expected_behavior`
- `covered_by_tc`
- `coverage_status`
- `gap_note`

### Правила колонок

- `atom_id` — стабильный локальный идентификатор атомарного утверждения в формате `ATOM-001`, `ATOM-002`, ... . Он является основным ключом связи между traceability matrix, review findings и writer response. Не переиспользуй `atom_id` для другого утверждения в следующем round.
- `req_id` — буквенно-цифровой код требования, если он есть. Если кода нет, используй стабильную ссылку на раздел. Не используй placeholder `-` или `N/A`.
- Если в подтвержденном scope есть буквенно-цифровой код требования, он должен появиться в `req_id` минимум одной строки matrix. Нельзя заменять его только `source_path`.
- Если `source-parity-check.md` содержит mandatory requirement ID или PDF-only requirement ID, этот код должен появиться в `req_id` минимум одной строки matrix.
- `source_path` — краткая иерархия источника: раздел, блок, поле или иной точный фрагмент ФТ.
- `atomic_statement` — одно неделимое утверждение из ФТ без перефразирования, меняющего смысл.
- `field_or_block` — поле, блок или сущность, к которой относится утверждение.
- `condition` — условие применимости правила. Если условия нет, укажи `always`.
- `expected_behavior` — какое поведение должно быть покрыто тестом.
- `covered_by_tc` — список `test_case_id`, покрывающих утверждение. Если покрытия нет, укажи `not_covered:<GAP-ID>` для `gap` или `unclear:<GAP-ID>` / `not_covered:unclear` для `unclear`; не используй placeholder `-` или `N/A`.
- `coverage_status` = `covered | gap | unclear`
- `gap_note` — краткое пояснение, почему строка имеет статус `gap` или `unclear`. Для `covered` используй `not_applicable:covered` или краткое подтверждение покрытия; не используй placeholder `-` или `N/A`.

## XLSX-дубль

- Для каждого Markdown artifact `round-N-traceability-matrix.md` создавай рядом файл `round-N-traceability-matrix.xlsx`.
- `.xlsx` должен содержать те же обязательные колонки и те же строки, что и Markdown matrix; не добавляй в `.xlsx` скрытых требований или дополнительных интерпретаций.
- Рекомендуемый лист для основной таблицы: `traceability`.
- Допустимо добавить отдельный лист `meta` с FT-пакетом, scope, round number и ссылкой на исходный Markdown artifact.
- Используй автофильтр, закрепленную строку заголовка и перенос текста в ячейках, чтобы матрица была пригодна для ручного анализа.
- Markdown остается каноническим текстовым artifact для diff/review, `.xlsx` является обязательным человекочитаемым дублем для анализа и фильтрации.

## Рекомендуемый шаблон

```md
## Traceability Matrix

| atom_id | req_id | source_path | atomic_statement | field_or_block | condition | expected_behavior | covered_by_tc | coverage_status | gap_note |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| ATOM-001 | GSR 11 | `2.1.1.1.1.1.2 -> Блок «Личная информация» -> Поле «ФИО клиента»` | Поле отображается всегда | Поле `ФИО клиента` | always | Поле видно пользователю | `TC-LI-001` | covered | not_applicable:covered |
| ATOM-002 | GSR 11 | `2.1.1.1.1.1.2 -> Блок «Личная информация» -> Поле «ФИО клиента»` | Поле обязательно, если `Ввести вручную = Нет` | Поле `ФИО клиента` | `Ввести вручную = Нет` | Пустое значение не принимается | `TC-LI-005` | covered | not_applicable:covered |
| ATOM-003 | `2.1.1.1.1.1.2` | `Блок «Личная информация» -> Поле «ФИО клиента»` | Ограничение по максимальной длине отсутствует в тексте, но макет не позволяет подтвердить поведение | Поле `ФИО клиента` | always | Требование недостаточно определено | unclear:GAP-001 | unclear | Нужна явная формулировка в основном ФТ |
```

## Правила использования

- Источник строк — только основной текст ФТ и связанные материалы этого же FT-пакета.
- Текст ФТ имеет приоритет над макетом.
- `source-parity-check.md` не расширяет scope, но является обязательной инвентаризацией кодов и строк источника при наличии DOCX+PDF.
- Reviewer не должен объединять разные правила в одну строку ради компактности.
- Writer не должен использовать матрицу для расширения scope.
- Findings по traceability должны ссылаться на строку матрицы через `traceability_ref = ATOM-*`, а не через текст `source_path` или повторяющийся `req_id`.
- Если строка матрицы переносится в следующий round без смыслового изменения, сохраняй тот же `atom_id`; если атомарное утверждение разделено на несколько независимых строк, старый `atom_id` не переиспользуй для новых смыслов и явно отметь split в human summary.
- `gap` означает отсутствие покрытия при достаточной определенности требования.
- `unclear` означает, что требование нельзя однозначно превратить в проверку без домысливания.

## Legacy Матрицы

Исторические `round-N-traceability-matrix.md`, созданные до введения обязательного `atom_id`, не нужно механически переписывать только ради соответствия этому формату. Такое переписывание создает ложную точность.

Если legacy matrix используется как baseline для нового writer/reviewer round, reviewer должен:

- явно отметить ее как `legacy traceability baseline` в human summary;
- при создании новой или обновленной matrix назначить стабильные `atom_id`;
- зафиксировать, как legacy rows сопоставлены с новыми `ATOM-*`, если это влияет на закрытие findings;
- не считать text-only ссылку на `req_id` или `source_path` достаточной для нового `traceability` finding.
