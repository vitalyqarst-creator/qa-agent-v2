# Test Case Format

Канонический формат ручного тест-кейса хранится в Markdown и должен быть пригоден для ручного выполнения и быстрой трассировки к ФТ.

Короткие пары плохих и хороших примеров по повторяемым стилевым ошибкам хранятся в `references/qa/test-case-style-examples.md`. Этот файл не заменяет правила ниже, а иллюстрирует их.

## Рекомендуемый шаблон

Canonical writer output must use the bold metadata/runtime fields shown below. Do not use table-only metadata (`| Поле | Значение |`) as the only representation of `Название`, `Тип`, `Приоритет`, `package_id`, or `Трассировка`; validator parsing treats that as missing required fields. Do not mix table metadata with duplicate bold fields either: the canonical form is bold fields only.

Runtime completeness has two canonical levels: the full recommended format below, and the slim runtime format defined in `test-case-runtime-format.md`. The slim format is valid only when it keeps required bold metadata/runtime sections complete; optional source fields are allowed when they add nonduplicative source navigation or real source evidence.

```md
## TC-DEMO-PI-001
**Название:** Обязательность поля «ФИО клиента» при значении «Ввести вручную = Нет»
**Приоритет:** High
**Тип:** Validation
**package_id:** `WP-01`

**Цель:** Проверить, что поле `ФИО клиента` обязательно к заполнению, если ручной ввод ФИО не включен.

**Предусловия:**
- Открыт раздел `Основная информация`.
- Переключатель `Ввести вручную` установлен в значение `Нет`.
- заполнены обязательные поля

**Тестовые данные:**
- Поле `ФИО клиента` оставлено пустым.

**Шаги:**
1. Оставить поле `ФИО клиента` пустым.
2. Нажать кнопку `Сохранить`

**Итоговый ожидаемый результат:** Сохранение не происходит. Поле `ФИО клиента` подсвечивается красным. Под полем появляется текст `Поле является обязательным для заполнения`

**Постусловия:**
- Нет специальных постусловий.

**Ссылка на ФТ:** `GSR 11`; `2.1.1.1.1.1.2`; PDF стр. 47

**Источник требования:**
- `2.1.1.1.1.1 Раздел «Основная информация»`
- `2.1.1.1.1.1.2 Описание свойств полей формы в разделе «Основная информация»`
- `Блок «Личная информация»`
- `Поле «ФИО клиента»`

**Источник / цитата требования:** Поле `ФИО клиента` отображается всегда и обязательно, если признак `Ввести вручную = Нет`.
```

## Example Specificity Warning

The exact red highlight and exact error text in the example are allowed only when they are explicitly stated by the FT, mockup, API spec, or approved package material. If the exact UI reaction is not defined, write the expected result at the verifiable behavior level, for example: `сохранение не происходит; изменение не применяется; значение не сохраняется после повторного открытия`. Do not use invisible judgment phrases such as `поле считается невалидным` as a standalone oracle. If pass/fail still depends on an undefined reaction, record a `coverage gap` / `unclear` instead of inventing details.

## Классы Наблюдаемого Oracle

`Итоговый ожидаемый результат` должен относиться минимум к одному наблюдаемому классу oracle. Если ни один класс нельзя вывести из ФТ/support artifacts, не создавай executable `TC-*`; фиксируй `GAP-*` / `unclear`.

Допустимые классы oracle для ручного TC:

- `field-state`: в конкретном поле отображается, очищается, изменяется или сохраняется конкретное значение.
- `transition-state`: конкретный раздел/экран открывается или не открывается после указанного действия.
- `save-state`: сохранение выполняется или не выполняется; после повторного открытия отображается конкретное сохраненное/несохраненное значение.
- `message-marker`: отображается конкретная source-backed подсказка, сообщение, marker обязательности или ошибка.
- `visibility-state`: конкретное поле, блок или действие отображается, скрыто, доступно или недоступно, если это задано источником.
- `list-state`: список/справочник содержит конкретные ожидаемые значения; отсутствие лишних значений проверяется только если закрытость списка следует из источника.
- `persistence-artifact`: UI/API/log/DB/mock artifact подтверждает внутренний результат, если такой artifact входит в scope и указан источником.
- `calculation-result`: результат расчета содержит формулу/source reference, входные значения и вручную вычисленный expected result.

Запрещено заменять oracle абстрактным verdict: `считается валидным`, `считается невалидным`, `не проходит валидацию`, `не соответствует правилу`, `соответствует ФТ`. Такие фразы допустимы только как пояснение рядом с конкретным наблюдаемым результатом, но не как основной pass/fail критерий.

## Структура кейса

- идентификатор тест-кейса;
- `Название`;
- `Приоритет`;
- `Тип`;
- `package_id`;
- `Цель`;
- `Предусловия`;
- `Тестовые данные`;
- `Шаги`;
- `Итоговый ожидаемый результат`;
- `Постусловия`;
- `Ссылка на ФТ`;
- `Источник требования`;
- `Источник / цитата требования`;

## Обязательные правила

- Один тест-кейс = одна проверка и один основной ожидаемый результат.
- Каждый `TC-*` должен иметь `package_id` из `scope-contract.md`; даже простой scope использует минимум один package, например `WP-01`.
- Один `TC-*` не должен смешивать разные `package_id`, кроме явно выделенного scenario/use-case package с отдельным обоснованием.
- `Ссылка на ФТ` и `Трассировка` не должны содержать placeholder `-`, `N/A`, пустой элемент между разделителями или другой фиктивный id. Если у исходного утверждения нет `GSR`/`REQ`, оставь только реальные `ATOM-*`, `SRC-*`, section/table/page/source quote/`DICT-*` references.
- Однотипная проверка нескольких значений одного элемента, списка или справочника считается одной проверкой, если все значения проверяются одним действием и одним ожидаемым результатом.
- Не дроби такую однотипную проверку на отдельные кейсы по одному значению только ради атомарности.
- Если отдельное значение запускает отдельную бизнес-логику, влияет на другие поля, доступность действий, валидацию или расчет, выделяй для него отдельный тест-кейс.
- Не используй параметризованный тест-кейс как замену нескольким самостоятельным executable scenarios. Если каждая строка таблицы требует отдельной подготовки состояния, отдельного запуска, отдельного pass/fail и отдельной трассировки к полю/атому, создавай отдельные `TC-*`, а не один кейс с `iteration`.
- Таблица в `Тестовые данные` допустима для сверки состава одного списка/справочника или набора однотипных значений, проверяемых одним действием и одним ожидаемым результатом; она не должна превращать один тест-кейс в batch-прогон нескольких независимых проверок.
- Если тест-кейс можно разделить на несколько независимых проверок, разделяй его на отдельные тест-кейсы.
- Не объединяй в одном тест-кейсе несколько разных проверок только ради экономии места.
- Не объединяй в одном тест-кейсе независимые свойства поля, такие как видимость, обязательность, редактируемость, формат, границы, default и persistence, если каждое свойство имеет отдельный pass/fail результат.
- Заголовок должен быть конкретным и проверяемым.
- Формулируй `Название` как короткое проверяемое утверждение с явным субъектом и действием: `Поле X блока Y принимает 10 цифр`, `Поле X блока Y отклоняет значение ниже 2000`, `Кнопка X открывает раздел Y`. Избегай отглагольных существительных и канцелярских конструкций вроде `Прием рабочего телефона из 10 цифр`, `Отклонение значения с длиной не 10 цифр`, если их можно заменить нормальной фразой `Поле ... принимает/отклоняет ...`.
- `Название` не должно представлять проверяемое условие как достаточную причину результата, если результат зависит от других валидных предусловий. Например, не пиши `Выбранный чекбокс X открывает раздел Y`, если раздел открывается только при валидности всей формы; пиши про проверяемую обязанность: `Выбранный чекбокс X удовлетворяет обязательности блока Y`.
- Если в scope есть несколько блоков с одинаковыми или однотипными полями, `Название` тест-кейса должно явно указывать блок, к которому относится проверка. Не заменяй название поля своим обобщающим термином вроде `основной доход`, если такого термина нет в ФТ; используй точное имя поля и блок-контекст, например: `Отображение поля «Среднемесячный доход после вычета налогов» в блоке «Основная работа» после заполнения «Тип занятости»`.
- Шаги должны быть воспроизводимыми.
- Итоговый ожидаемый результат формулируется как наблюдаемое состояние системы, достаточное для pass/fail, а не только как перефразирование названия тест-кейса.
- Трассировка должна указывать на конкретный источник требования.
- Если в шаблоне используется цитата требования, она должна помогать идентифицировать исходное правило, а не подменять собой всю трассировку.

## Правила организации набора

- В одном canonical test-case file группируй `TC-*` по функциональности, блоку, элементу формы, бизнес-операции или внутреннему work package, к которому они относятся. Выбирай самый конкретный уровень, который следует из подтвержденного scope и помогает читать набор без смешивания независимых областей.
- Группировка не должна расширять scope, объединять независимые проверки в один `TC-*` или заменять трассировку к ФТ.
- Внутри группы сохраняй локальный порядок проверок для элемента: базовые свойства, состав списков/справочников, позитивные проверки, негативные проверки.
- Не перезапускай нумерацию внутри групп. Числовой суффикс `TC-*` должен быть сквозным в пределах файла: `...-001`, `...-002`, `...-003` без дублей, пропусков и нескольких `...-001` в разных группах.
- Если после review меняется группировка или порядок кейсов, writer выполняет финальную нормализацию перед повторным review: переставляет кейсы, перенумеровывает `TC-*` и обновляет все ссылки на них в ledger, coverage map, applicability/risk/pairwise/dependency tables, traceability matrix и writer response artifacts текущего round.
- После `signed-off` не выполняй тихую перенумерацию. Любое изменение `TC-*` id после sign-off является новой revision и требует повторного review-cycle.

## Правила заполнения полей

- `Приоритет` и `Тип` фиксируются явно, если набор кейсов ведется в расширенном формате.
- `Предусловия` описывают стартовое состояние перед выполнением кейса; действие, которое проверяется тест-кейсом, не должно быть предусловием этого же кейса.
- `Предусловия` должны фиксировать только состояние, значимое для проверки: этап, статус, открытую форму, заполненность ключевых полей или наличие нужной сущности. Если состояние создано, выбрано или настроено в UI (`создан блок`, `задано условие`, `настроено состояние`), укажи воспроизводимый путь: шаг пользователя с кнопкой/полем/значением/блоком или fixture/setup artifact. Если путь неизвестен из ФТ, mockup, UI notes или handoff, оформи `GAP-*` / `unclear`, а не executable TC.
- Для baseline-состояния формы используй естественные ручные формулировки, например `Остальные обязательные поля заполнены валидными значениями`. Не пиши канцелярское `заполнены валидно`, если можно описать состояние через значения полей.
- Формулировки валидного baseline вроде `Минимальный валидный набор данных`, `валидная заявка`, `валидные данные` и `остальные поля заполнены валидно` допустимы только когда исполнитель может однозначно восстановить нужные поля и значения из шагов кейса, конкретных предусловий, таблицы fixture в том же артефакте или явно связанного разрешенного artifact. Если baseline нельзя воспроизвести без догадки, это placeholder: раскрой конкретное состояние или оформи `GAP-*` / `unclear`.
- Не указывай `fixture_id` в `Тестовые данные` ручного TC, если он не является самостоятельным входным артефактом проверки. Если fixture нужна только как shorthand для валидного baseline, описывай baseline через исполнимые предусловия, например `Остальные обязательные поля заполнены валидными значениями`; `fixture_id` оставляй в design artifacts или используй в TC только когда без него подготовка состояния становится неоднозначной.
- `Тестовые данные` должны содержать только специальные входные данные или характеристики подготовленной сущности, нужные для выполнения проверки. Если специальных данных нет, указывай `Нет специальных тестовых данных`; не повторяй в этом поле шаги или проверяемое действие.
- `Тестовые данные` не должны заменять конкретный проверяемый ввод фразой про generic baseline: `Минимальный валидный набор данных...`, `валидное значение`, `валидные данные для формы`, `значение из тестовых данных`. Для validation/equivalence/boundary кейса укажи конкретный literal, параметризованное значение с source (`P_min - 1`, `P_max + 1`) или `GAP-*`, если конкретный ввод нельзя выбрать из разрешенных материалов.
- Не переноси в `Тестовые данные` состояние, которое полностью создается шагами. Если кейс проверяет выбор значения, включение переключателя или установку чекбокса, эти действия должны быть в `Шаги`, а `Тестовые данные` нужны только для независимого входного значения, fixture или заранее подготовленной сущности.
- Не заполняй `Тестовые данные` значениями, которые полностью создаются шагами и не важны как конкретные входные данные. Если значение можно заменить на `любой валидный` / `любое значение из справочника` без изменения pass/fail, оставь это в шаге или предусловии, а в `Тестовые данные` укажи `Нет специальных тестовых данных`.
- Не добавляй test-design class в ручные `Тестовые данные`, если конкретное значение уже задано. Например, вместо `Сумма: 2000,5; класс: decimal-separator` пиши только `2000,5`; class храни в TDDT/plan/coverage artifacts.
- Не используй в `Тестовые данные` англоязычные design labels и branch shorthand вроде `Dictionary values`, `Selected tag`, `Branch state`, `condition=false`, если это не literal из источника. Для ручного TC пиши русское исполнимое состояние или конкретное входное значение.
- Если шаги уже однозначно указывают поле и блок, `Тестовые данные` могут содержать только само значение. Не повторяй имя поля и блок в данных без необходимости.
- Если тестовые данные описывают подготовленную сущность, фиксируй только свойства данных, которые делают ее пригодной для проверки; не дублируй все предусловия без необходимости. Конкретное значение поля задавай только если оно влияет на ожидаемый результат; иначе достаточно указать значимое состояние данных, например `поле заполнено`.
- `Постусловия` заполняются даже если специальных постусловий нет; в этом случае указывай `Нет специальных постусловий`.
- `Трассировка` должна содержать реальные source-link tokens: `ATOM-*`, requirement code, `SRC-*`, `DICT-*`, section/table/page/source locator. Это обязательное поле связи TC с источником.
- `Ссылка на ФТ` может содержать код требования, номер раздела, страницу PDF или другой краткий навигационный набор, если это добавляет информацию сверх `Трассировка`.
- `Источник требования` может показывать иерархию до нужного блока, поля или правила, если эта иерархия не очевидна из `Трассировка`.
- Если одноименное поле встречается в нескольких блоках, `Источник требования` должен включать путь до конкретного блока, а не только раздел и поле. Если текст ФТ и макет называют блок по-разному, используй название из ФТ как каноническое, а UI-название из макета добавляй как alias только для снятия неоднозначности.
- `Источник / цитата требования` должен содержать краткую реальную цитату без расширения scope либо явно помеченную нормализацию/derived rationale. Не выдавай synthetic statement или пересказ test-design obligation за цитату ФТ.
- Не дублируй один и тот же набор source tokens в `Ссылка на ФТ`, `Трассировка` и `Источник требования`. Если optional source field не добавляет новой навигации или evidence, не включай его.

## Правила приоритизации

`Приоритет` должен отражать риск проверяемого атома, а не удобство исполнения или порядок в документе.

- `High`: деньги, права доступа, потеря или порча данных, необратимые state transitions, критичные расчеты, обязательные бизнес-валидации, server-side rejection, безопасность, интеграции с внешним эффектом.
- `Medium`: основные пользовательские сценарии, частые валидации, зависимости полей, списки/таблицы, сохранение и повторное открытие данных, типовые negative cases без критичного ущерба.
- `Low`: редкие UI-состояния, вспомогательные проверки, косметические состояния и низкорисковые варианты, которые не меняют бизнес-результат.

Если атом относится к high-risk domain, у него должен быть хотя бы один `High` test case или blocking `coverage gap`. Не понижай приоритет только потому, что сценарий редко встречается, если ущерб от дефекта высокий.

## Правила для шагов и ожидаемого результата

- Шаги должны быть детерминированными. Не используй формулировки `если форма еще не открыта`, `при необходимости`, `если требуется`, если это не является проверяемой веткой сценария.
- Описывай действие в шагах на уровне намерения пользователя, а не способа взаимодействия с UI. Для текстового поля используй формулировку `Ввести <значение> в поле <название>`. Для поля даты используй `Указать <дату> в поле <название>`. Не используй `щелкнуть поле`, `кликнуть поле`, `нажать на поле` и `набрать`, если сам факт активации/фокуса не является отдельным source-backed поведением. Если проверяется именно реакция поля на активацию, пиши `Активировать поле <название>` без привязки к мыши.
- Если состояние зависит от ветки сценария, перенеси его в предусловия или выдели отдельный тест-кейс.
- В негативном тест-кейсе шаг не должен утверждать успешное выполнение запрещенного действия. Формулируй его как попытку: `попытаться изменить`, `попытаться сохранить`, `попытаться выбрать`, если именно запрет является предметом проверки.
- Не указывай конкретный тип UI-контрола или способ взаимодействия, если он не следует из ФТ, макета или разрешенных материалов пакета. Используй нейтральную формулировку вроде `попытаться изменить значение поля доступным в интерфейсе способом`.
- Если тип UI-контрола следует из ФТ, макета или разрешенных материалов пакета, называй его явно в шагах: `в переключателе выбрать значение Да`, `выбрать чекбокс`, `нажать кнопку`, `ввести значение в поле`. Не заменяй известный control generic формулировкой `установить поле`.
- В шагах и источнике требования используй точное имя поля и блок-контекст из ФТ или разрешенных материалов, если они известны. Не заменяй их кратким alias вроде `поле суммы дополнительного дохода`, когда можно написать `поле «Среднемесячный доход после вычета налогов» блока «Дополнительный доход»`.
- Не используй placeholder-контекст `в соответствующем блоке`, `в нужном блоке`, `в применимом блоке`, если блок известен из ФТ, mockup inventory или source artifacts. Назови конкретный блок; если блок неизвестен и это мешает исполнению, фиксируй `GAP-*` / `unclear`.
- Если для UI scope есть `mockup-visual-inventory.md`, шаги должны использовать его для точных названий видимых блоков, actions и interaction hints. Не заменяй это generic формулировками вроде `перейти к нужному элементу` или `ввести или выбрать значение`, если inventory показывает конкретный способ взаимодействия. При этом mockup не является источником бизнес-правил: allowed values, обязательность, validation и expected result должны следовать из ФТ/source artifacts, иначе фиксируй `GAP-*` / `unclear`.
- Итоговый ожидаемый результат должен описывать конкретный проверяемый эффект: открытая форма, отображаемый список, доступность действия, сообщение, состояние поля или результат валидации.
- Если переход используется как observable oracle, указывай целевой раздел/статус/экран. Если переход зависит от валидности всей формы, expected result должен явно сохранять этот контекст, например: `значение X принимается; при валидно заполненных остальных полях раздела открывается раздел Y`.
- `Итоговый ожидаемый результат` должен быть настолько коротким, насколько позволяет pass/fail. Не повторяй полное имя поля и блока, если они уже однозначно заданы в шагах; для negative transition допустимо: `Переход заблокирован: раздел Y не открывается; значение X не принимается`. Не добавляй вторичный oracle вроде `пользователь остается в разделе X`, если `раздел Y не открывается` уже однозначно подтверждает блокировку и источник не требует отдельной проверки текущего раздела.
- Не используй `считается валидным`, `считается невалидным`, `валидируется`, `не проходит валидацию` как итоговый expected result без наблюдаемого эффекта. Укажи, что именно видит или может проверить исполнитель: отображаемое значение поля, блокировку перехода/сохранения, неизменившееся значение после повторного открытия, конкретное сообщение, если оно source-backed.
- Не используй oracle `значение из тестовых данных принято`, `значение из тестовых данных не принимается` или аналогичную ссылку на test data, если в `Тестовые данные` нет конкретного literal/параметра/строки fixture. Expected result должен называть проверяемое значение или наблюдаемый эффект так, чтобы pass/fail был понятен без обратного вывода из generic baseline.
- Если ФТ задает ограничение ввода или формата для конкретного поля, это ограничение определяет допустимый/недопустимый класс значений, но не выбирает UI-механизм enforcement. К таким ограничениям относятся: `только N цифр`, `только числовые символы`, `только текстовые символы`, `маска`, `максимальная длина`, `точная длина`, `допустимые/недопустимые символы`.
- Для input restrictions writer должен выбрать один подтвержденный observable oracle. Допустимые варианты: field-state (`в поле остается значение <literal>` / `символ <literal> не добавляется`) только если это подтверждено ФТ, mockup, UI evidence или утвержденными материалами; transition-state (`переход в раздел Y заблокирован`) только при полном valid fixture остальных обязательных полей; message-marker только с source-backed marker/message. Если механизм enforcement не подтвержден, создай узкий `GAP-*` / `unclear`, а не executable baseline TC.
- Не используй альтернативный oracle для одного negative input TC: `символ отклонен или значение осталось пустым/предыдущим`, `значение очищено или поле подсвечено`, `не сохранено или не принято` и похожие варианты не дают детерминированного pass/fail.
- Не проверяй field-level input restrictions через `Сохранить`, `Следующий шаг` или другую transition validation без полного воспроизводимого valid fixture для всех остальных обязательных полей выбранной ветки. Исключение: значение короче точной длины или пустое обязательное поле может требовать validation action, потому что поле не может запретить пользователю прекратить ввод раньше лимита.
- В ручном TC называй конкретный введенный символ обычным языком: `пробел`, `буква A`, `символ #`, `запятая`, `минус`. Не подменяй это канцелярскими формулировками вроде `пробельный символ`, `буквенный символ`, `недопустимый символ из ввода`; не называй запятую `десятичным разделителем`, если десятичная семантика не следует из источника.
- Не используй `обычное состояние`, `обычное поле`, `без маркера заполнения` как oracle, если source-backed marker или control state не определены. Для обратной ветки обязательности проверяй конкретный наблюдаемый результат: поле не требует заполнения при source-backed действии перехода/сохранения, либо поле отображается/не отображается, если именно это задано ФТ.
- Не помещай неопределенность в итоговый expected result: фразы вроде `способ реакции UI не уточнен источником`, `точная реакция не определена`, `реализация может...` означают, что baseline TC не имеет детерминированного oracle. Зафиксируй это как `GAP-*` / `unclear` или выбери подтвержденный observable oracle.
- Для date-dependent правил, завязанных на `текущую дату`, тест должен оставаться исполнимым в любой день. Используй относительное значение (`текущая дата - 18 лет`, `завтра`) либо явно зафиксируй управляемую бизнес/системную дату в предусловиях, если тестовая среда позволяет ее задать или проверить. Не клади `Дата проверки` только в `Тестовые данные`, если исполнитель не может управлять этой датой; не используй прошедшую абсолютную дату как `текущую`.
- Для default value TC в `Название` и expected result указывай фактическое ожидаемое значение, если оно есть в ФТ: `отображает значение по умолчанию Да/Нет`. Не пиши машинные остатки вроде `значение по умолчанию значение по умолчанию из строки ФТ`; если само значение в источнике не извлечено, фиксируй `GAP-*` / `unclear`, а не создавай TC с placeholder.
- Сырой текст ФТ, `GSR`/`REQ` quote или строка normalization не является итоговым ожидаемым результатом. Не пиши `Для <поле> наблюдается правило из GSR N: ...`; перенеси такой текст в `Источник / цитата требования`, а в expected result оставь конкретный observable oracle.
- Не используй source-rule oracle: `по правилу из источника`, `по правилу видимости из источника`, `согласно источнику`, `согласно ФТ`, `по требованиям ФТ`. Это не pass/fail результат. Для visibility укажи конкретно `поле отображается`, `поле не отображается`, `блок доступен`, `кнопка недоступна` и условие, созданное шагами.
- Если в source extraction есть разорванные слова или table residue (`Д а`, `П о`, `Рефинансировани е`, `Переключат ель`, `Название Видимость О Р Тип ввода поля Тип значения Примечание`), не копируй это в steps/expected result. Сначала нормализуй источник или оформи `GAP-*` / `unclear`.
- Не добавляй точные значения, тексты ошибок, статусы, форматирование или UI-реакции, если они не следуют из ФТ или разрешенных материалов пакета.
- Если точная UI-реакция не задана, ожидаемый результат должен оставаться проверяемым без домысливания: например, `значение не изменяется` или `на текущем этапе отсутствует доступный способ выполнить действие`, без утверждения про disabled-состояние, текст ошибки или тип контрола.
- Не используй в итоговом ожидаемом результате размытые формулировки: `либо`, `согласно реализации`, `согласно системному правилу`, `если бизнес запрещает`, `корректно обрабатывается`, `не ломается`, `должно быть уточнено`. Если без такой оговорки нельзя сформулировать pass/fail, не пиши baseline-кейс; зафиксируй `coverage gap` / `unclear`.
- Не используй альтернативные expected results через `или`, если варианты являются разными наблюдаемыми реакциями. Формулировка `значение очищено, не сохранено или поле подсвечено ошибкой` не дает детерминированного pass/fail; выбери один подтвержденный oracle или вынеси точную реакцию в `GAP-*` / `unclear`.
- Если полезная проверка основана на переходе состояния, зависимости, восстановлении после ошибки или регрессионном риске, но ожидаемая реакция не задана ФТ или разрешенным уточнением, не превращай ее в baseline-кейс по предположению.
- Для numeric/input rejection запрещено чинить неподтвержденный field-filtering oracle другим неподтвержденным oracle. Если источник говорит только `не принимает значение`, `только цифры`, `значение ниже минимума не принимается` или аналогичный class rule, это не доказывает `поле подсвечено красным`, `появляется ошибка`, `Следующий шаг заблокирован`, `раздел не открыт`, `значение очищено` или `значение не отображается`. Такие UI reactions допустимы только при прямом source/UI/support evidence именно для invalid numeric/input class. Иначе сохраняй invalid class в `Coverage Obligation Table` / design artifacts и фиксируй enforcement mechanism как narrow `GAP-*` / `unclear`.
- `тип значения: Строка/Дата/Логическое` сам по себе не создает отдельный baseline TC. Закрывай такую строку через конкретный source-backed input/widget/format behavior или `GAP-*`; не пиши standalone TC с oracle `поле принимает и отображает введенное строковое значение`, если это только metadata.
- Для внутренних или интеграционных результатов expected result должен называть наблюдаемый artifact проверки. Нельзя считать покрытыми сохранение `kladr`, `esiaUserId`, `CorrelationId`, API/RabbitMQ calls, async waiting/retry, DB/model persistence или отсутствие downstream calls только по формулировке ФТ, если не указан UI/API/log/DB/RabbitMQ/mock artifact для pass/fail.
- Для пользовательски видимых полей сохранение/несохранение можно проверять без DB/API/log artifact, если ФТ задает save/no-save outcome и есть in-scope путь повторно открыть тот же объект/раздел. Expected result должен называть конкретное отображаемое значение после повторного открытия или отсутствие несохраненного значения.
- Если FT описывает и пользовательское действие, и внутренний результат, но artifact есть только для действия, выделяй action-initiation кейс отдельно, а внутренний результат фиксируй как `coverage gap` / `unclear`.
- Для расчетного кейса expected result должен содержать formula/source reference, входные значения и вручную вычисленный expected result. Если rounding, precision, валюта или единицы измерения влияют на pass/fail и не описаны источниками, фиксируй `coverage gap` / `unclear`.

## Правила для списков и справочников

- Если ФТ задает несколько допустимых значений одного поля, списка или справочника и проверка для всех значений одинакова, формируй один тест-кейс на состав списка, а не отдельный кейс на каждое значение.
- В таком кейсе перечисляй все ожидаемые значения и указывай все связанные requirement id / atomic statement id.
- Для отображаемых значений проверяй точное написание из ФТ: регистр, кавычки, дефисы, пробелы, числовые значения, проценты, сокращения и коды.
- Если ФТ задает закрытый перечень значений, ожидаемый результат должен проверять и наличие ожидаемых значений, и отсутствие лишних значений.
- Для support workbook / `dictionary-inventory.md` с active values формулируй expected как `отображаются все и только активные значения DICT-*`; если есть archived/inactive values, явно проверяй их отсутствие или фиксируй `GAP-*`, если UI не позволяет надежно отличить active/inactive.
- Если из ФТ не следует, что перечень закрытый, не добавляй проверку отсутствия лишних значений по предположению; при необходимости фиксируй это как `coverage gap` или `unclear`.
- Если отдельное значение меняет дальнейшую бизнес-логику, доступность полей, доступность действий, валидацию или расчет, покрывай это отдельным тест-кейсом по соответствующему правилу ФТ.

## Правила для editability TC

- Editability TC проверяет возможность изменить значение конкретного элемента, а не просто факт клика/фокуса.
- Указывай исходное значение и новое значение в предусловиях, тестовых данных или шагах. Новое значение должно быть literal из ФТ, справочника, fixture или разрешенного источника.
- Для справочника/selector формулируй шаги через действие пользователя: `открыть список`, `выбрать значение <literal>`; для текстового поля: `ввести <literal>`; для переключателя/checkbox: `установить значение <literal>`.
- Запрещены связки `Активировать элемент` + `Изменить значение на тестовое значение`, если TC не раскрывает, какое именно значение выбрано/введено и что должно отображаться после изменения.
- Итоговый ожидаемый результат должен содержать конкретное новое значение, например `в поле отображается значение <literal>`. Если нужно доказать persistence, это отдельный save/reopen TC или явно названный основной oracle.

## Test-design Derived Checks

- Проверки, появившиеся из test-design анализа, defect taxonomy, reviewer feedback, exploratory/experience-based слоя или risk model, полезны, но не являются новыми строками ФТ.
- В таких TC и design artifacts явно помечай источник как `test-design-derived`, `experience-based`, `risk-derived` или `v2 obligation` и указывай `source_or_rule_ref`: defect class, coverage dimension, reviewer finding, risk row или accepted canary lesson.
- Не записывай derived check как обычный source-backed `ATOM-*` без объяснения derivation. Если TC ссылается на исходный `SRC-*`, формулировка должна показывать, какую source-backed обязанность он уточняет, а не расширяет.
- Derived checks не должны подменять baseline FT coverage: сначала должны быть покрыты прямые source-backed atoms или оформлены `GAP-*`.

## Рекомендуемый порядок для поля

Если набор тестов относится к одному конкретному полю, придерживайся такого порядка:

- сначала проверка видимости поля;
- затем проверка редактируемости поля;
- затем проверка значения по умолчанию и обязательности, если это описано в ФТ;
- затем проверка состава списка или справочника, если поле выбирает значение из перечня;
- потом атомарные позитивные проверки;
- потом атомарные негативные проверки.

Не смешивай в одном кейсе:

- позитивные и негативные проверки;
- независимые валидационные сценарии;
- разные причины отклонения значения;
- отклонение недопустимого значения и прием допустимого значения;
- несколько классов эквивалентности с разными pass/fail результатами.

Для ограничений ввода применяй отдельное правило разделения классов:

- проверка `значение не принимается` и проверка `значение принимается` являются разными проверками;
- недопустимый класс и допустимый класс нельзя закрывать одним итоговым ожидаемым результатом вида `не принимает X и принимает Y`;
- допустимое значение можно использовать в негативном кейсе только как setup/precondition, если оно не является частью итогового expected result; если нужно доказать recovery после ошибки, создай отдельный `Recovery`/`Scenario` TC с основным oracle про восстановление;
- один негативный тест-кейс проверяет один недопустимый класс, если классы можно различить по ФТ или по правилам тест-дизайна;
- для length/mask validation не объединяй значения ниже требуемой длины и выше допустимой длины в один executable case, если для них возможны разные наблюдаемые реакции: например, короткое значение может вводиться и отклоняться при переходе дальше, а длинное значение может не вводиться, обрезаться или блокироваться маской;
- допустимое значение проверяй отдельным позитивным тест-кейсом, если оно нужно для покрытия;
- если после ошибки нужно проверить восстановление вводом допустимого значения, оформляй это как отдельный восстановительный сценарий и явно указывай, что основной ожидаемый результат — восстановление после ошибки.
- `Тестовые данные` для validation/equivalence/boundary не должны быть placeholders: запрещены формулировки `значение, нарушающее правило`, `значение из проверяемого класса`, `дата, соответствующая ветке ФТ`, `не требуются`, если pass/fail зависит от конкретного ввода.
- Если точное значение нельзя выбрать без внешнего справочника, не подставляй произвольное значение: оформи `GAP-*` или укажи параметризованный класс с источником, например `P_min - 1`, `P_min`, `P_max + 1`, где `P_min/P_max` берутся из подтвержденного продуктового каталога.
- Для action/async case ожидаемый результат не может ограничиваться фразой `действие инициируется`. Укажи конкретный наблюдаемый результат или artifact; если его нет в scope, перенеси assertion в `GAP-*` / `unclear`.

## Запрещенные Подмены Покрытия

Не закрывай metadata или неисполняемую подготовку формальным `TC-*`:

- `тип значения: Строка`, `тип значения: Дата`, `тип значения: Логическое Да/Нет` не доказываются открытием списка и выбором значения. Такой TC допустим только если источник прямо говорит, что поле берет значение из списка/справочника/перечня. Иначе покрывай наблюдаемый input/format/control behavior или фиксируй `GAP-*`.
- Не создавай `TC-*` только для фиксации gap/unclear. Если строка source является metadata-only или не имеет standalone observable oracle, она должна быть отражена в `Coverage Gaps`, ledger, Package Test Design Plan или traceability matrix, но не как executable `## TC-*`.
- Dependency setup должен быть действием пользователя с конкретным управляющим элементом и значением. Запрещен шаг `В форме задать данные, при которых ...` без названия controlling field, exact value and action.
- Expected result не должен быть generic rule-oracle: `видимое состояние после действия соответствует одному наблюдаемому правилу`, `поле изменяет, отображает или ограничивает значение`, `поле считается невалидным`. Укажи конкретный observable state или `GAP-*`.
- В `Тестовые данные` поле `класс` должно описывать класс, а не второе значение. Правильно: `Допустимое значение: 123456; класс: 6 digits`. Неправильно: `Допустимое значение: 123456; класс: 12345`.

## Правила трассировки

Если в ФТ есть код требования:

- указывай код требования, например `GSR 22`, `REQ-15`.
- код требования должен ссылаться на то же исходное утверждение, которое проверяет кейс. Нельзя использовать соседний `GSR`/`REQ` как формальную ссылку, если поле, condition или expected behavior отличаются.

Если кода требования нет:

- указывай раздел, блок, поле или точную формулировку.

Источник требования должен помогать быстро найти исходное место в документе, но не превращаться в громоздкую техническую вставку.
