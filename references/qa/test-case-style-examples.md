# Test Case Style Examples

Этот reference хранит короткие пары `Bad` / `Good` для повторяемых ошибок оформления ручных тест-кейсов. Канонические правила остаются в `references/qa/test-case-format.md`; этот файл только иллюстрирует их.

Добавляй сюда только повторяемые паттерны, а не разовые вкусовые правки. Если пример специфичен для одного FT-пакета, фиксируй его в package notes или review artifact, а не в общем reference.

## Предусловия: fixture-id вместо исполнимого состояния

**Rule:** `test-case-format.md` -> `Правила заполнения полей`

**Bad**

```md
- Применена fixture `VALID_EMPLOYMENT_HIRE_BASE`.
```

**Good**

```md
- Остальные обязательные поля заполнены валидными значениями.
```

**Why**

`fixture_id` полезен в design artifacts, но ручной тест-кейс должен быть понятен исполнителю без восстановления fixture. Используй `fixture_id` в TC только если без него подготовка состояния становится неоднозначной.

## Предусловия: magic state вместо setup steps

**Rule:** `test-case-format.md` -> `Правила заполнения полей`

**Bad**

```md
### Предусловия

У поля `Адрес регистрации` отображается подсказка о квартире.
```

**Good**

```md
### Предусловия

1. Авторизоваться в системе.
2. Открыть карточку «Заявка» с доступным блоком «Адреса клиента».
3. Перейти в блок «Адреса клиента».
4. Если переключатель `Ввести вручную` для адреса регистрации находится не в значении `Нет`, установить значение `Нет`.
5. В поле `Адрес регистрации` ввести адрес с регионом и домом, но без квартиры.
6. Выбрать найденную подсказку DaData для введенного адреса, если это требуется текущим режимом поля.
7. Дождаться отображения подсказки `Укажите номер квартиры или поставьте отметку о проживании в частном доме`.
```

**Why**

Голое UI-состояние не говорит manual executor или automation agent, как подготовить систему. Если состояние создается fixture/API, назови fixture/API явно; если создается UI-действиями, опиши последовательность setup steps внутри `Предусловия`.

## Тестовые данные: test-design class в ручном TC

**Rule:** `test-case-format.md` -> `Правила заполнения полей`

**Bad**

```md
- Сумма: `2000,5`; класс: `decimal-separator`.
```

**Good**

```md
- `2000,5`
```

**Why**

Класс нужен для TDDT, plan, coverage или reviewer analysis. Ручному исполнителю достаточно конкретного значения, если поле и действие уже заданы в шагах.

## Тестовые данные: baseline fixture вместо проверяемого значения

**Rule:** `test-case-format.md` -> `Правила заполнения полей`

**Bad**

```md
**Тестовые данные:**
- Валидный базовый набор `VALID_EMPLOYMENT_ADDITIONAL_INCOME`: остальные поля набора заполнены валидными значениями; проверяемое поле изменяется в шагах.
- `12 34`
```

**Good**

```md
**Предусловия:**
- Добавлен блок «Дополнительный доход».
- Поле «Тип дохода» заполнено валидным значением.
- Остальные обязательные поля заполнены валидными значениями.

**Тестовые данные:**
- `12 34`
```

**Why**

Baseline и fixture setup относятся к предусловиям или design artifacts. В `Тестовые данные` ручного TC оставляй только проверяемое входное значение, если шаги уже задают, куда его вводить.

## Тестовые данные: повторение поля и блока

**Rule:** `test-case-format.md` -> `Правила заполнения полей`

**Bad**

```md
**Тестовые данные:**
- Проверяемое значение для поля «Среднемесячный доход после вычета налогов» в блоке «Дополнительный доход»: `2000,5`.

**Шаги:**
1. Ввести `2000,5` в поле «Среднемесячный доход после вычета налогов» блока «Дополнительный доход».
```

**Good**

```md
**Тестовые данные:**
- `2000,5`

**Шаги:**
1. Ввести `2000,5` в поле «Среднемесячный доход после вычета налогов» блока «Дополнительный доход».
```

**Why**

Если шаг уже однозначно указывает поле и блок, `Тестовые данные` должны оставаться минимальными.

## Шаги: alias поля вместо точного имени и блока

**Rule:** `test-case-format.md` -> `Правила для шагов и ожидаемого результата`

**Bad**

```md
1. Ввести `12 34` в поле суммы дополнительного дохода.
```

**Good**

```md
1. Ввести `12 34` в поле «Среднемесячный доход после вычета налогов» блока «Дополнительный доход».
```

**Why**

Краткий alias может быть удобен в разговоре, но ручной TC должен однозначно указывать поле. Если точное имя поля и блок известны, используй их в шагах и источнике требования.

## Шаги: mouse-specific ввод в поле

**Rule:** `test-case-format.md` -> `Правила для шагов и ожидаемого результата`

**Bad**

```md
1. щелкнуть поле `Сумма на руки` и набрать `123`
```

**Good**

```md
1. Ввести `123` в поле `Сумма на руки`.
```

**Why**

Ручной TC должен описывать проверяемое намерение, а не способ фокусировки контрола. `Щелкнуть` привязывает тест к мыши без source-backed причины, а `набрать` хуже стандартизовано, чем `Ввести`.

## Предусловия: канцелярское baseline-состояние

**Rule:** `test-case-format.md` -> `Правила заполнения полей`

**Bad**

```md
- Остальные обязательные поля заполнены валидно.
```

**Good**

```md
- Остальные обязательные поля заполнены валидными значениями.
```

**Why**

Формулировка должна звучать как состояние, которое может подготовить ручной исполнитель.

## Expected Result: избыточное повторение поля

**Rule:** `test-case-format.md` -> `Правила для шагов и ожидаемого результата`

**Bad**

```md
**Итоговый ожидаемый результат:** Переход заблокирован: раздел «Анкета клиента» не открывается; значение `2000,5` не принимается как корректное значение поля «Среднемесячный доход после вычета налогов» в блоке «Дополнительный доход».
```

**Good**

```md
**Итоговый ожидаемый результат:** Переход заблокирован: раздел «Анкета клиента» не открывается; значение `2000,5` не принимается.
```

**Why**

Поле уже задано в шагах. Expected result должен оставаться коротким, если pass/fail не становится двусмысленным.

## Expected Result: лишний текущий раздел

**Rule:** `test-case-format.md` -> `Правила для шагов и ожидаемого результата`

**Bad**

```md
**Итоговый ожидаемый результат:** Переход заблокирован: раздел «Анкета клиента» не открывается; пользователь остается в разделе «Сведения о занятости»; значение `12 34` не принимается.
```

**Good**

```md
**Итоговый ожидаемый результат:** Переход заблокирован: раздел «Анкета клиента» не открывается; значение `12 34` не принимается.
```

**Why**

Если целевой раздел не открывается, блокировка перехода уже наблюдаема. Не добавляй отдельный oracle про текущий раздел, если источник не требует проверить именно это состояние.

## Expected Result: невидимая невалидность

**Rule:** `test-case-format.md` -> `Правила для шагов и ожидаемого результата`

**Bad**

```md
**Итоговый ожидаемый результат:** Значение с недопустимым символом из ввода `12 3` считается невалидным для поля `Сумма на руки` по правилу «только цифровые символы».
```

**Good**

```md
**Итоговый ожидаемый результат:** В поле отображается значение `123`; пробел из введенного значения `12 3` не отображается.
```

**Good: validation source-backed only when FT explicitly defines validation on action**

```md
**Итоговый ожидаемый результат:** Переход заблокирован: раздел «Анкета клиента» не открывается; значение `12 3` не принимается.
```

**Why**

`Считается невалидным` не является наблюдаемым результатом. Для ограничений ввода и формата конкретного поля проверяй отображаемое значение поля после попытки ввода, если ФТ явно не задает другой механизм проверки.

## Expected Result: символ сверх точной длины

**Rule:** `test-case-format.md` -> `Правила для шагов и ожидаемого результата`

**Bad**

```md
1. Ввести `1234567` в поле `Код подразделения`.
2. Нажать кнопку `Следующий шаг`.

**Итоговый ожидаемый результат:** Переход заблокирован: раздел «Сведения о занятости» не открывается; значение `1234567` не принимается.
```

**Good**

```md
1. Ввести `1234567` в поле `Код подразделения`.

**Итоговый ожидаемый результат:** В поле `Код подразделения` отображается значение `123456`; седьмая цифра `7` не добавляется.
```

**Why**

Если ФТ задает `только 6 цифр` / `содержит 6 цифр`, проверка значения сверх лимита выполняется на уровне поля. Transition validation допустима только если источник отдельно задает проверку на переходе/сохранении.

## Название/Цель: техническое название символа

**Rule:** `test-case-format.md` -> `Правила для шагов и ожидаемого результата`

**Bad**

```md
**Название:** `Сумма на руки` не принимает пробельный символ в цифровом вводе
**Цель:** Подтвердить запрет пробельного символа в поле `Сумма на руки`.
```

**Good**

```md
**Название:** `Сумма на руки` не принимает пробел в цифровом вводе
**Цель:** Проверить, что поле `Сумма на руки` не принимает пробел при вводе цифрового значения.
```

**Why**

Для ручного исполнителя важен конкретный ввод и наблюдаемый результат, а не техническая классификация символа. Пиши `пробел`, `буква A`, `символ #`, `запятая`, `минус`; не используй канцелярские формулировки вроде `пробельный символ` или `недопустимый символ из ввода`.

## Шаги/Expected Result: placeholder-контекст блока

**Rule:** `test-case-format.md` -> `Правила для шагов и ожидаемого результата`

**Bad**

```md
1. Посмотреть состояние тумблера «Ввести вручную» в соответствующем блоке раздела.

**Итоговый ожидаемый результат:** Поле «Регион» отображается в соответствующем блоке.
```

**Good**

```md
1. Посмотреть состояние тумблера «Ввести вручную» в блоке «Адрес регистрации».

**Итоговый ожидаемый результат:** Поле «Регион» отображается в блоке «Адрес регистрации».
```

**Why**

`Соответствующий блок` не помогает ручному исполнителю найти элемент. Если блок известен из ФТ, mockup inventory или support artifact, называй его явно; если неизвестен и это влияет на исполнение, фиксируй `GAP-*` / `unclear`.

## Expected Result: неопределенная реакция UI

**Rule:** `test-case-format.md` -> `Правила для шагов и ожидаемого результата`

**Bad**

```md
**Итоговый ожидаемый результат:** Значение с лишней цифрой считается невалидным; способ реакции UI на лишнюю цифру не уточнен источником.
```

**Good**

```md
**Итоговый ожидаемый результат:** Переход заблокирован: раздел «Сведения о занятости» не открывается; значение `12345` не принимается.
```

**Why**

Expected result должен быть детерминированным. Если единственное честное утверждение — `реакция не уточнена`, это не baseline TC; вынеси вопрос в `GAP-*` / `unclear` или проверь source-backed действие валидации.

## Date-Dependent TC: текущая дата в тестовых данных

**Rule:** `test-case-format.md` -> `Правила для шагов и ожидаемого результата`

**Bad**

```md
**Тестовые данные:**
- Дата проверки: `2026-06-01`.
- Дата рождения: `2008-06-01`.
```

**Good**

```md
**Предусловия:**
- Тестовая среда позволяет зафиксировать бизнес-дату проверки `2026-06-01`.

**Тестовые данные:**
- Дата рождения: `2008-06-01`.
```

**Good: dynamic**

```md
**Тестовые данные:**
- Дата рождения: текущая дата минус 18 лет плюс 1 день.
```

**Why**

Если правило зависит от текущей даты, абсолютные даты быстро устаревают. Фиксируй управляемую дату в предусловиях или используй относительное значение, иначе кейс перестает быть исполнимым и трассируемым.

## Название: default value placeholder

**Rule:** `test-case-format.md` -> `Правила для шагов и ожидаемого результата`

**Bad**

```md
**Название:** «Визуальная информация» отображает значение по умолчанию значение по умолчанию из строки ФТ
```

**Good**

```md
**Название:** «Визуальная информация» отображает значение по умолчанию `Нет`
```

**Why**

Название должно содержать проверяемое значение, а не след машинной нормализации. Если значение по умолчанию не извлечено из источника, оформляй `GAP-*` / `unclear`, а не baseline TC с placeholder.

## Expected Result: обычное состояние в обратной ветке

**Rule:** `test-case-format.md` -> `Правила для шагов и ожидаемого результата`

**Bad**

```md
**Итоговый ожидаемый результат:** Поле остается обычным полем без маркера заполнения в обратной ветке условия.
```

**Good**

```md
**Итоговый ожидаемый результат:** Переход не блокируется из-за пустого поля «Город» при заполненном поле «Населенный пункт».
```

**Why**

`Обычное состояние` и `маркер заполнения` не являются надежным oracle без source-backed UI marker. Для обратной ветки обязательности проверяй конкретное поведение: отсутствие блокировки перехода/сохранения, видимость поля или другое подтвержденное состояние.

## Название: условие выдано за достаточную причину

**Rule:** `test-case-format.md` -> `Обязательные правила`

**Bad**

```md
**Название:** При выбранном параметре визуальной оценки открывается раздел «Анкета клиента»
```

**Good**

```md
**Название:** Выбранный параметр визуальной оценки удовлетворяет условию минимального выбора
```

**Why**

Если переход зависит от валидности всей формы, не представляй одно условие как достаточную причину открытия следующего раздела.

## Шаги: неизвестный тип UI-элемента

**Rule:** `test-case-format.md` -> `Правила для шагов и ожидаемого результата`

**Bad**

```md
1. Установить «Визуальная информация» = `Да`.
```

**Good**

```md
1. В переключателе «Визуальная информация» выбрать значение `Да`.
```

**Why**

Если тип control известен из ФТ, макета или support artifact, называй его явно.
