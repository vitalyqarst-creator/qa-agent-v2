# Traceability Rules

## Общие правила

- Каждый кейс должен ссылаться на конкретный раздел или фрагмент требования.
- Если есть буквенно-цифровой код требования, указывай именно его.
- Если кода нет, указывай раздел, поле, блок или точную формулировку.
- Буквенно-цифровой код требования, попавший в подтвержденный scope, нельзя опускать из ledger или traceability matrix. Если поведение по коду нельзя покрыть тест-кейсом, оно должно быть зафиксировано как `gap` или `unclear`, а не исчезнуть из трассировки.
- Если `source-parity-check.md` нашел код требования только в PDF, этот код считается попавшим в подтвержденный scope и обязателен для `req_id`.

## Приоритеты

- Текст ФТ важнее макета.
- DOCX остается главным исходным документом ФТ / source of truth.
- XHTML обязателен как основной машиночитаемый источник извлечения требований для таблиц, списков, вложенных списков, перечней значений и source rows.
- Если в FT-пакете есть PDF-версия основного ФТ, используй ее для сверки структуры разделов, заголовков и границ scope.
- PDF используется именно для structural cross-check и навигации, а не как замена текста основного ФТ.
- Если DOCX и PDF доступны одновременно, перед writer/reviewer handoff должен быть создан `source-parity-check.md` по `references/agent/source-parity-check-format.md`. Этот artifact определяет обязательные к сохранению requirement IDs, включая PDF-only коды.
- Support-файлы и макеты помогают интерпретации, но не заменяют текст основного ФТ.
- Если source refs из ledger/matrix/TC ссылаются на table/list-heavy content, они должны сохранять ссылку на XHTML row/list fragment, когда именно XHTML дал полное machine-readable extraction.

## Формат трассировки

- Используй краткую, но информативную иерархию источника.
- Не дублируй всю иерархию документа, если это не помогает найти требование.
- Не выделяй `Коды требований` в отдельный верхнеуровневый раздел результата.

## Atomic Requirements Ledger

- Перед написанием новых тест-кейсов разложи подтвержденный scope на `atomic requirements ledger`: одна строка = одно атомарное утверждение ФТ или одно отдельное правило.
- Каждому atomic statement назначай стабильный локальный идентификатор `atom_id`, например `ATOM-001`, сохраняя буквенно-цифровой код требования, если он есть.
- Один requirement id может соответствовать нескольким atomic statements. Если требование содержит несколько самостоятельных проверяемых утверждений, не объединяй их в один тест-кейс только из-за общего кода требования.
- Каждый буквенно-цифровой код требования внутри scope должен иметь минимум одну строку `ATOM-*` или строку traceability matrix. Если код содержит несколько самостоятельных утверждений, создай несколько строк с тем же `req_id`.
- Все IDs из секции `Mandatory Traceability Inputs` в `source-parity-check.md` должны иметь минимум одну строку `ATOM-*` или строку traceability matrix.
- Каждый тест-кейс должен ссылаться на один или несколько atomic statement id либо на точный код требования, если atomic ledger уже встроен в traceability matrix.
- Если `req_id` есть в ledger/matrix, каждый покрывающий `TC-*` должен сохранять трассировку к этому `req_id` напрямую или через связанный `ATOM-*`.
- Review findings по трассировке должны ссылаться на atomic statement через `traceability_ref = ATOM-*`; не используй один только requirement id как ключ для исправления gap.
- Не объединяй независимые свойства одного поля в один atomic statement: видимость, редактируемость, обязательность, значение по умолчанию, формат, ограничения, зависимости и реакции системы должны быть отдельными строками, если они независимо описаны в ФТ.
- Для условной видимости разделяй позитивную и негативную ветки на отдельные atomic statements, если обе ветки следуют из ФТ: элемент отображается при выполнении условия и элемент не отображается при невыполнении условия. Если отрицательная ветка не определена однозначно, фиксируй `coverage gap` или `unclear`, а не добавляй предполагаемое поведение.
- Если expected behavior нельзя вывести из ФТ и разрешенных материалов пакета, не превращай его в тест-кейс; фиксируй `coverage gap` или `unclear`.
- Не добавляй точные значения, разделители, статусы, тексты ошибок, порядок нормализации или UI-реакции, если они не следуют из ФТ или разрешенных материалов пакета.
